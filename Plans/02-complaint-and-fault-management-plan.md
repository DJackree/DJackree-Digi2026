# Complaint And Fault Management Plan

## Purpose

This plan covers the Complaint and Fault Management module. It defines customer complaint submission, agent work queues, admin assignment/reporting, workflow enforcement, notes, status history, and SLA tracking.

## Module Sections

- Complaint records and categories.
- Complaint workflow and status transitions.
- Customer complaint portal.
- Agent work queue.
- Admin complaint management.
- Notes and resolution history.
- SLA and dashboard reporting.

## Section 1 - Complaint Records And Categories

### Phase 0 - Design

Purpose:

- Capture customer issues with enough structure to support tracking, assignment, workflow, reporting, and chatbot queries.

Entities:

- `Complaint`
- `ComplaintNote`
- `ComplaintStatusHistory`
- `CustomerAccount`
- Agent `User`

Business rules:

- A complaint must belong to one customer account.
- A complaint starts as `Open`.
- A complaint may be unassigned at creation.
- Internal notes are never visible to customers.
- Resolution notes are visible to agents/admins and may be summarized to customers only if intentionally exposed.

Assumptions:

- Categories are fixed choices rather than a database-managed table to reduce implementation time.
- Complaint reference numbers are generated by the system and displayed to users.

Design decisions:

- Use model choices for category and status because requirements list a stable, small set.
- Use separate `ComplaintNote` records instead of one large notes field so updates are auditable.

### Phase 1 - Database

Recommended model sketch:

```python
class Complaint(models.Model):
    class Category(models.TextChoices):
        BILLING = "billing", "Billing"
        NETWORK = "network", "Network"
        DEVICE = "device", "Device"
        ROAMING = "roaming", "Roaming"
        OTHER = "other", "Other"

    class Status(models.TextChoices):
        OPEN = "open", "Open"
        IN_PROGRESS = "in_progress", "In Progress"
        ESCALATED = "escalated", "Escalated"
        RESOLVED = "resolved", "Resolved"
        CLOSED = "closed", "Closed"

    reference = models.CharField(max_length=20, unique=True, db_index=True)
    customer_account = models.ForeignKey("customers.CustomerAccount", on_delete=models.CASCADE, related_name="complaints")
    category = models.CharField(max_length=30, choices=Category.choices, db_index=True)
    description = models.TextField()
    status = models.CharField(max_length=30, choices=Status.choices, default=Status.OPEN, db_index=True)
    assigned_agent = models.ForeignKey(User, null=True, blank=True, on_delete=models.SET_NULL, related_name="assigned_complaints")
    escalation_reason = models.TextField(blank=True)
    resolved_at = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True, db_index=True)
    updated_at = models.DateTimeField(auto_now=True)
```

Field rationale:

- `reference`: user-friendly tracking identifier.
- `customer_account`: ties complaint to account ownership and chatbot context.
- `category`: supports dashboard grouping.
- `description`: captures user-provided issue details.
- `status`: supports workflow and reporting.
- `assigned_agent`: supports agent queue.
- `escalation_reason`: captures why escalation occurred.
- `resolved_at`: simplifies average resolution time calculation.
- timestamps: support sorting, aging, SLA, and audit.

Recommended notes model:

```python
class ComplaintNote(models.Model):
    complaint = models.ForeignKey(Complaint, on_delete=models.CASCADE, related_name="notes")
    author = models.ForeignKey(User, on_delete=models.PROTECT)
    body = models.TextField()
    is_internal = models.BooleanField(default=True)
    created_at = models.DateTimeField(auto_now_add=True)
```

Recommended status history model:

```python
class ComplaintStatusHistory(models.Model):
    complaint = models.ForeignKey(Complaint, on_delete=models.CASCADE, related_name="status_history")
    changed_by = models.ForeignKey(User, on_delete=models.PROTECT)
    from_status = models.CharField(max_length=30, blank=True)
    to_status = models.CharField(max_length=30)
    note = models.TextField(blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
```

Constraints and indexes:

- Index `status`, `category`, `created_at`, and `assigned_agent`.
- Use `on_delete=PROTECT` for note/history authors to preserve audit data.
- Use `SET_NULL` for assigned agent so old complaints remain if an agent is deleted.

Migration notes:

- Create complaint tables after `customers` and `accounts`.
- Add a helper to generate unique references such as `CMP-2026-0001`.

### Phase 2 - API And Routes

Customer routes:

| Method | URL | Purpose |
|---|---|---|
| GET | `/complaints/` | List current customer's complaints |
| GET | `/complaints/new/` | Show complaint form |
| POST | `/complaints/new/` | Create complaint |
| GET | `/complaints/<reference>/` | View complaint detail |

Agent routes:

| Method | URL | Purpose |
|---|---|---|
| GET | `/agent/complaints/` | List assigned complaints sorted by age |
| GET | `/agent/complaints/<reference>/` | View assigned complaint |
| POST | `/agent/complaints/<reference>/status/` | Move complaint forward |
| POST | `/agent/complaints/<reference>/notes/` | Add internal note |
| POST | `/agent/complaints/<reference>/escalate/` | Escalate with reason |

Admin routes:

| Method | URL | Purpose |
|---|---|---|
| GET | `/admin-portal/complaints/` | List all complaints |
| GET | `/admin-portal/complaints/<reference>/` | View any complaint |
| POST | `/admin-portal/complaints/<reference>/assign/` | Assign or reassign agent |
| POST | `/admin-portal/complaints/<reference>/status/` | Change to any valid status |

Example create payload if implemented as JSON or form fields:

```json
{
  "category": "network",
  "description": "Mobile data has been unavailable since this morning."
}
```

Example status update payload:

```json
{
  "status": "in_progress",
  "note": "Customer contacted. Investigating network signal in area."
}
```

Authentication and authorization:

- All routes require login.
- Customer querysets filter by `customer_account.user == request.user`.
- Agent querysets filter by `assigned_agent == request.user`.
- Admin querysets do not restrict by assignment.

Validation:

- Category is required on creation.
- Description is required and should have a reasonable minimum length.
- Agent status update must be a valid forward transition.
- Escalation requires an escalation reason.
- Admin assignment must target a user with agent role.

Error handling:

- Invalid form input returns form errors.
- Unauthorized object access returns 404 from scoped querysets.
- Invalid workflow transition returns form error or 400 for JSON.

### Phase 3 - UI

Customer pages:

- Complaint history table with reference, category, status, submitted date, last updated.
- Complaint creation form with category and description.
- Complaint detail page showing customer-visible status timeline and latest status.

Agent pages:

- Assigned complaint queue sorted oldest first.
- Filters for status and category.
- Detail page with customer/account summary, description, internal notes, status form, escalation form.

Admin pages:

- All complaints table with filters by status, category, agent, SLA status.
- Detail page with assignment form, status override form, full notes/history.
- Link to admin dashboard metrics.

Role-based behavior:

- Customers cannot see assigned agent controls, internal notes, or status forms.
- Agents can update only assigned complaints.
- Admins can assign complaints and use broader status controls.

Key interactions:

- Flash success message after create/update.
- Inline validation errors for forbidden transitions.
- Dashboard links from metric cards to filtered complaint lists.

## Section 2 - Workflow Enforcement

### Phase 0 - Design

Purpose:

- Centralize complaint state transition rules so views, forms, and future tests use the same logic.

Allowed lifecycle:

```text
Open -> In Progress -> Escalated -> Resolved -> Closed
```

Agent rules:

- Agent can move only forward.
- Agent can escalate assigned complaint with reason.
- Agent cannot close a complaint unless it is already resolved.

Admin rules:

- Admin can move complaint to any listed status.
- Admin can reopen by moving backward only if business needs arise; for this assessment, "any status" is allowed by requirement.

Customer rules:

- Customer cannot change status.

### Phase 1 - Database

Workflow state is stored on `Complaint.status`.

Status history is stored in `ComplaintStatusHistory`.

Set `Complaint.resolved_at` when:

- New status is `resolved`.
- `resolved_at` is currently null.

Do not clear `resolved_at` if an admin later changes status for demonstration simplicity, unless the implementation explicitly supports reopening.

### Phase 2 - API

Create service functions:

```python
def get_allowed_statuses(user, complaint):
    ...

def change_complaint_status(*, complaint, new_status, changed_by, note=""):
    ...

def assign_complaint(*, complaint, agent, assigned_by):
    ...
```

Validation examples:

- Customer calling status service raises permission error.
- Agent on unassigned complaint raises permission error.
- Agent trying `open -> resolved` raises validation error if strict single-step forward movement is enforced.
- Admin status changes are allowed if target status is valid.

### Phase 3 - UI

For agents:

- Render only allowed next actions as buttons.
- Require note field when marking resolved or escalating.

For admins:

- Render dropdown with all statuses.
- Show warning text that admin overrides are audited.

## Section 3 - SLA And Dashboard Reporting

### Phase 0 - Design

Purpose:

- Provide the required admin dashboard metrics and show operational visibility.

Required metrics:

- Total complaints by status.
- Total complaints by category.
- Average resolution time from Open to Resolved.
- Complaints breaching SLA threshold.

Business rules:

- SLA threshold is 5 calendar days from `created_at`.
- Resolved and closed complaints are not currently breaching SLA.
- Average resolution uses complaints with `resolved_at IS NOT NULL`.

### Phase 1 - Database

No separate SLA table is needed. Use complaint timestamps and status.

Helpful indexes:

- `status`
- `category`
- `created_at`
- `resolved_at`

### Phase 2 - API And Routes

Admin dashboard route:

| Method | URL | Purpose |
|---|---|---|
| GET | `/admin-portal/dashboard/` | Show complaint metrics |

Dashboard query service:

```python
def get_dashboard_metrics():
    return {
        "by_status": ...,
        "by_category": ...,
        "average_resolution_time": ...,
        "sla_breaches": ...,
    }
```

SLA query:

```python
Complaint.objects.exclude(
    status__in=[Complaint.Status.RESOLVED, Complaint.Status.CLOSED]
).filter(
    created_at__lt=timezone.now() - timedelta(days=5)
)
```

### Phase 3 - UI

Dashboard components:

- Metric cards for total open, in progress, escalated, resolved, closed.
- Category distribution table.
- Average resolution time card.
- SLA breach table with reference, customer, category, age, assigned agent.

Interactions:

- Metric cards link to filtered complaint list.
- SLA breach rows link to complaint detail.

## Section 4 - Fault/Outage Records

### Phase 0 - Design

Purpose:

- Support chatbot answers about active faults or outages affecting a customer's area.

Entity:

- `NetworkOutage`

Business rules:

- Outages are tied to a region.
- Active outages are those with `is_active=True`.
- Customers can ask whether outages affect their area through the chatbot.

Assumptions:

- Outages are seeded/static for the assessment.
- No full outage management UI is required unless time remains.

### Phase 1 - Database

```python
class NetworkOutage(models.Model):
    region = models.CharField(max_length=100, db_index=True)
    title = models.CharField(max_length=200)
    description = models.TextField()
    started_at = models.DateTimeField()
    estimated_resolution_at = models.DateTimeField(null=True, blank=True)
    is_active = models.BooleanField(default=True, db_index=True)
    created_at = models.DateTimeField(auto_now_add=True)
```

Field rationale:

- `region`: connects outage to customer profile/account.
- `title` and `description`: explain the fault.
- timestamps: support realistic outage status.
- `is_active`: simple active/inactive filtering.

### Phase 2 - API

No customer-facing outage API is required for the first version. The chatbot context builder should query:

```python
NetworkOutage.objects.filter(region=customer.region, is_active=True)
```

Admin can manage outages through Django admin if time is limited.

### Phase 3 - UI

Minimum UI:

- Register `NetworkOutage` in Django admin.

Optional custom UI:

- Admin outage list and create/edit pages.
